# BIMInspect — Project Context

## What this project does

Automated detection of concrete surface damage in inspection photos, with a primary focus on **bridge and infrastructure concrete**. A fine-tuned YOLO11m model detects 5 damage classes and writes every detection into an IFC BIM model as an `IfcAnnotation` with `Pset_DamageInspection`.

Training data is dominated by bridge/infrastructure concrete imagery from CODEBRIM (European bridge facade dataset) and Roboflow public concrete defect datasets.

## Goals

1. **Automate damage surveys** — replace manual walkthroughs with camera footage + AI.
2. **Geo-reference every defect** — convert pixel bounding boxes into 3-D building coordinates.
3. **Enrich the BIM model** — attach damage severity, type, area, and photos as IFC properties.
4. **Generate inspection reports** — produce PDF/HTML summaries directly from IFC data.

## Pipeline overview

```
[Input images / video frames]
        │
        ▼
┌───────────────────┐
│  1. YOLO11m       │  src/detection/
│  Tiled Inference  │  • 640×640 sliding window, stride 480
│                   │  • Cross-tile NMS (torchvision)
│                   │  • Outputs bounding boxes, class labels, confidence scores
└────────┬──────────┘
         │  detections (pixel coords + metadata)
         ▼
┌───────────────────┐
│  2. Coordinate    │  src/pipeline/
│  Mapping          │  • Camera intrinsics / extrinsics → ray casting
│                   │  • OR: known reference markers → homography
│                   │  • Pixel bbox  →  3-D world XYZ  →  IFC local coords
└────────┬──────────┘
         │  damage objects with 3-D position + extent
         ▼
┌───────────────────┐
│  3. IFC Export    │  src/bim/
│                   │  • Opens template IFC from ifc/templates/
│                   │  • Creates IfcAnnotation per defect
│                   │  • Attaches Pset_DamageInspection (type, confidence, bbox…)
│                   │  • Saves enriched model to ifc/output/
└────────┬──────────┘
         │
         ▼
┌───────────────────┐
│  4. Reporting     │  src/utils/  +  results/
│                   │  • JSON / CSV detections → results/detections/
│                   │  • Annotated images      → results/visualizations/
│                   │  • PDF / HTML reports    → results/reports/
└───────────────────┘
```

## Damage classes

| ID | Class | Description |
|---|---|---|
| 0 | `crack` | Hairline to structural cracks in concrete |
| 1 | `spallation` | Concrete surface loss / spalling |
| 2 | `efflorescence` | Salt deposits / moisture ingress |
| 3 | `exposed_bars` | Visible reinforcement bars |
| 4 | `corrosion` | Rebar / steel rust staining |

## Dataset (v15-rf — current)

Training data: **CODEBRIM auto-labeled tiles + 5 Roboflow concrete/bridge defect datasets**.
No manual labels — removed after testing showed they added noise without sufficient diversity.
Val set is 20% split from the same sources, so scores reflect real performance across all domains.

| Source | Images | Domain |
|---|---|---|
| CODEBRIM auto-tiled | ~10,000 tiles | European bridge/building facades |
| Roboflow: Concrete 5November | 7,443 | Mixed concrete infrastructure |
| Roboflow: corrosion (3 severities) | 5,473 | Bridge/industrial corrosion |
| Roboflow: Efflorescence-Det | 1,011 | Concrete walls |
| Roboflow: Exposure-Det | 1,032 | Exposed rebar, bridge decks |
| Roboflow: Spalling-Det | 1,009 | Concrete spalling |

**build_dataset.py flags:**
- `USE_ROBOFLOW = True`
- `USE_MANUAL = False`
- 80% train / 20% val split on all sources

## Model training history

| Version | mAP@50 | Key change |
|---|---|---|
| **v15-rf** | **62.3%** | YOLO11m, CODEBRIM + Roboflow, val includes Roboflow 20% — **current best** |
| v16 | ~60% peak, declining | Fine-tune from v15-rf best.pt, lr0 too high, abandoned |
| v14 | ~33% plateau | CODEBRIM + Manual, 8-way augmentation caused slow decline |
| v13 | declining from ~33% | Added Roboflow but val was CODEBRIM-only → domain mismatch |
| v12 | 33.0% | Manual labels only, too few images |
| v10 | 57.7% | CODEBRIM auto-labels, YOLOv8m — previous production |
| v11 | 51.4% | Domain mixing with crack tiles hurt performance |
| v9 | 57.8% | YOLOv8s tiled inference |
| v7 | 57.5% | First tight XML bbox training |
| v4 | 99.4% | Crack only, single class |

## Development notes

- **Primary domain:** bridge and infrastructure concrete — model performs best on close-up facade/bridge photos similar to CODEBRIM and Roboflow training data.
- CODEBRIM zip has a 4 GB offset bug: every image entry's `header_offset` is inflated by 2³² bytes. Fix: retry at `header_offset − 4294967296` when PK magic not found.
- Model weights go in `models/weights/` — git-ignored (large files).
- IFC templates in `ifc/templates/` are read-only — always write enriched copies to `ifc/output/`.
- `detector.py` supports detection mode (`best_detection.pt`) and classification fallback (`best.pt`) with Grad-CAM bbox estimation.
- Tiled inference: `TILE_SIZE=640`, `TILE_STRIDE=480`, cross-tile NMS via `torch.ops.torchvision.nms`.
- When fine-tuning from a checkpoint, use `lr0=0.001` (not 0.01) to avoid destabilising trained weights.
